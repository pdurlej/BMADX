# BMADX Repository Audit — 2026-07-10

Scope: Codex CLI compatibility, GPT-5.6 Sol/Terra/Luna readiness, BMAD gate
health, benchmark integrity, goal/loop semantics, tests, and agent handoff.

## Executive Read

BMADX has a sound core: the `X1..X4` classifier, soft/hard BMAD gate split,
rare `X4`, deterministic local tests, and benchmark artifact naming are all
worth keeping.

The failures came from drift around that core:

- the operator CLI was one version below the GPT-5.6 minimum,
- the skill and installer still described GPT-5.5 as the only tuned lane,
- benchmark thinking values and claims were not isolated by model capability,
- goal guidance did not distinguish interactive `/goal` from `codex exec`,
- BMAD dependency drift initially blocked `X3/X4`,
- project context lagged behind the released version,
- no CI protected the repository.

The result ships as `v0.3.0`. GPT-5.6 profiles are implemented as
canary-passed compatibility candidates, not promoted performance or default
model claims.

## Findings

### P0 — GPT-5.6 could not work on the installed CLI

Evidence:

- observed Codex CLI before remediation: `0.143.0`,
- OpenAI minimum for GPT-5.6 in Codex: `0.144.0`,
- local CLI was upgraded to `0.144.1`,
- local ChatGPT desktop/Codex mode is `26.707.31428`, above the OpenAI minimum
  `26.707.30751`,
- `check_codex_compat.py --require-gpt56` now passes for Sol, Terra, and Luna.

Impact: model selection could fail before BMADX activation, making skill-level
changes irrelevant.

Status: fixed in the operator environment; repo now documents and checks the
minimum.

Source: [OpenAI GPT-5.6 availability](https://help.openai.com/en/articles/20001354).

### P0 — BMAD dependency drift blocked `X3/X4`

Evidence:

- the initial live BMADX compact check allowed `X2` but denied `X3` execution,
- global BMAD was synchronized to upstream `v6.10.0`,
- BMADX then synchronized its dependency baseline,
- the final live X3 compact check allows classification and execution with
  `bmad_status=ok`.

Impact: red-zone and Rescue Mode work appears broken even when model and BMADX
code are healthy.

Decision: do not soften the gate. `X3/X4` blocking was the intended safety
contract. The dependency and BMADX baseline were refreshed under explicit
operator approval.

Status: fixed in the operator environment; backups were retained in
`/private/tmp` for rollback.

### P1 — GPT-5.5 was hardcoded across product and benchmark surfaces

Evidence:

- installer copy, README, skill manifest, benchmark runner default, and
  performance verifier default all named GPT-5.5,
- benchmark runs could start without an explicit model.

Impact: silent model drift, misleading installation output, and summaries that
look comparable when they are not.

Status: fixed. Benchmark model is now required, verifier filtering is opt-in,
and summaries include the selected model profile.

### P1 — Reasoning validation did not match the GPT-5.6 catalog

Observed local catalog on Codex CLI `0.144.1`:

| Model | Default | Supported reasoning |
| --- | --- | --- |
| Sol | `low` | `low`, `medium`, `high`, `xhigh`, `max`, `ultra` |
| Terra | `medium` | `low`, `medium`, `high`, `xhigh`, `max`, `ultra` |
| Luna | `medium` | `low`, `medium`, `high`, `xhigh`, `max` |

The old validator accepted `minimal` and rejected `max/ultra`.

Impact: valid GPT-5.6 output could fail validation, while an unsupported Codex
effort could be described as canonical.

Status: fixed. Runtime catalog capability and BMADX recommendation policy are
now separate. `max/ultra` are recognized but never automatic defaults.

### P1 — Performance claims could compare different models

Evidence: claim grouping used only BMAD profile and group slug. A fixed Sol run
and advisor Terra run could be paired.

Impact: invalid token/latency claims.

Status: fixed. Coverage and fixed/advisor pairing are isolated per model.

### P1 — Goal mode could loop on approvals or hard stops

Evidence:

- previous contract required a generic stop condition but did not require a
  blocked/approval termination path,
- benchmark validation treated any `Loop: yes` as bounded,
- interactive `/goal` syntax was presented as if it also dispatched through
  `codex exec`.

Impact: repeated blocked continuation turns, quota burn, and false confidence
that a headless goal was created.

Status: fixed. Goal lines require a success or blocked stop condition; loop
lines require a numeric bound and stop condition; headless creation requires an
explicit natural-language request.

Sources: [Codex improvement-loop use case](https://learn.chatgpt.com/use-cases/iterate-on-difficult-problems), [Codex goal control-plane clarification](https://github.com/openai/codex/issues/26949).

### P2 — Benchmark source paths ignored `CODEX_HOME`

Evidence: auth metadata and BMAD dependency paths were read from `~/.codex`
even when a custom `CODEX_HOME` was active.

Impact: custom-profile systems could use the wrong dependency or fail to find
auth metadata.

Status: fixed.

### P2 — Active context and automation had drifted

Evidence:

- `_bmad-output/project-context.md` still described `v0.2.9`,
- public release was `v0.2.10`,
- no GitHub Actions workflow existed.

Status: fixed in `v0.3.0` with a current project context and local-test CI.

## Model Decision

The profile policy is deliberately asymmetric:

- Sol: primary candidate for `X3/X4` and final synthesis; test `high` before
  escalating Rescue execution to `xhigh`.
- Terra: balanced candidate for everyday work; keep `X4=xhigh` until evidence.
- Luna: fast `X1/X2` candidate; do not promote `X3/X4` until red-zone and
  overwrite-safety evidence exists.

OpenAI describes Sol as the most capable, Terra as the balanced lower-cost
option, and Luna as fastest/lowest cost. The system card also reports stronger
complex-edit conflict avoidance and correctness for larger profiles, which is
why model equality would be the wrong abstraction.

Source: [GPT-5.6 Preview System Card](https://deploymentsafety.openai.com/gpt-5-6-preview/gpt-5-6-preview.pdf).

## Verification Performed

- all pre-change local tests: `96` passing,
- post-change deterministic suite: `106` passing,
- tests cover model profiles, catalog drift, minimum CLI, independent goal/loop
  selection, blocked stop conditions, bounded loops, and cross-model claim
  isolation,
- local model catalog compatibility passes on Codex CLI `0.144.1`,
- BMAD `v6.10.0` and the live `X3` hard gate pass after synchronization,
- subscription-backed healthy canaries pass baseline verification for Sol,
  Terra, and Luna,
- Sol retains total-token cap warnings in the `X1/X2` core cases,
- Luna retains them in `X1/X2` core plus pricing-copy and onboarding-email
  cases; Terra has none in its canary.

The first Sol canary correctly exposed that the policy did not say clearly
enough that a multi-turn goal does not imply a repair loop. The skill and
benchmark prompt were tightened, covered by a regression assertion, and the
same Sol canary then passed.

## Residual Risks

- `tokens used` parsing still depends on the human-readable Codex footer and
  should eventually migrate to a stable JSON event contract,
- Sol and Luna total-token warnings need repeated measurement before any
  efficiency recommendation,
- model policy is provisional until repeated healthy/degraded evidence exists.

## Recommended Next Move

Run the repeated healthy/degraded matrix from `docs/bmadx-v0.3-plan.md` before
promoting any model scope or publishing token, latency, cost, or default-model
claims. Prioritize Sol/Luna token stability and Sol degraded goal/loop
behavior.
